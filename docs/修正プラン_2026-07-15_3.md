## 設計方針（#9・#10・#12・#13を1本にまとめる）

#9（進軍・解禁の色分けログ）、#10（機能していない解禁モーダルの削除）、#12（手渡し済み表示が今季以外も出る）、#13（謁見の結果欄の削除）は、すべて「結果フェーズに“今季起きたこと”だけを見せる」という同じ目的でつながっている。バラバラに直すと矛盾するので、**共通の「今季イベントログ」機構**を新設し、まとめて解決する。

### 新設する仕組み
```js
gs.turnEvents = gs.turnEvents || []; // {text, color} の配列。今季分のみ保持
function pushTurnEvent(text, color){ // color: 'warn'(黄) | 'danger'(赤・新設) | 'info' | 'accent'
  gs.turnEvents = gs.turnEvents || [];
  gs.turnEvents.push({text:text, color:color});
}
```
- `goToNextTurn()`の**冒頭**（ターン加算の直前）で`gs.turnEvents = [];`にリセットする（前ターン分を残さないため）。
- CSSに`.log-danger{color:#c0392b}`のような赤字クラスを追加（既存`.log-ok/.log-warn/.log-accent/.log-info`と並べる。L243付近）。

### 各発生源をこの仕組みに接続する
1. **進軍イベント（黄）**: `advanceNodeAfterClear()`（L9374付近）の`showToast(...'ノード'+next+'へ進みました')`（L9415）の直後に、`pushTurnEvent(PNAMES[key]+'が'+NODE_LOC[next]+'へ進軍！', 'warn')`を追加。showToast自体は残してよい（結果ログとは別に一瞬の通知があっても害はない）。
2. **解禁イベント（赤）**: `markNodeCleared`ラッパー（L3778-3787付近）内の`queueUnlock('unlock_cart_fee', ...)`等の呼び出しを、**そのままqueueUnlockは残しつつ**（#10で最終的に消すため、まずは重複でよい）、同じ条件分岐に`pushTurnEvent('荷車が解禁された！\n利用には金20両かかりますが、積載2で運べます。', 'danger')`のように赤字版を追加する。boat/wagon解禁も同様。
3. **進言解放イベント（赤）**: 現状「解放された瞬間」を検知するイベント発火点がない。`refreshFacilityUnlockCaps()`（L5912付近）の呼び出し時に、直前の値と比較して0→1（またはn→n+1）に変わった進言があれば`pushTurnEvent('（皇子名）の進言「〇〇」が選べるようになった！', 'danger')`を出す。比較対象は`smithStaffMax/pharmStaffMax/marketLvMax/stableExtraMax`の4種。ただし`refreshFacilityUnlockCaps`は1ターン中に何度も呼ばれるため、**変化検知は1回だけ**にする必要がある（例：`gs._facilityMaxSnapshot`をターン開始時に保存し、結果フェーズ入場時に1回だけ比較する）。
4. **表示側**: `buildResultScreen()`（L2039付近）で、既存の`lines.push(...)`ループの中に`(gs.turnEvents||[]).forEach(function(ev){ lines.push({t:ev.text, c: ev.color==='danger'?'log-danger':ev.color==='warn'?'log-warn':ev.color}); });`を追加する。

## #14 信頼度の前季差分表示

- 市場前期比（`gs.market.prevBuy`）と同じパターンを踏襲する。
- `goToNextTurn()`の**外側ラッパーのうち、ターン送り前に前季値を保存できる最初のフック**（#2の調査で判明した市場と同様、ターン送りのコア処理より前で保存する必要がある）で、`gs.trustPrev = Object.assign({}, gs.trust);`を保存する。具体的には、既存の市場ラッパー（L8481付近「⑤ ターン進行: 前期保存 → 季節在庫 → 今季価格」IIFE）と同じ`goToNextTurn`ラップパターンを流用し、そのtry節冒頭に追記するのが安全（新しいラッパーを1つ追記してもよい）。

  ```js
  const prevT = (gs.trustPrev && gs.trustPrev[key]!=null) ? gs.trustPrev[key] : t;
  const diff = t - prevT;
  const diffText = diff===0 ? '' : (diff>0 ? '（+' + diff + '）' : '（' + diff + '）');
  ```

- 初回ターン（`gs.trustPrev`未設定）は差分なし（空文字）でよい。

## #16 市場価格：seasonPriceDelta以外の値を排除

- 静的コード調査済み：実行時に最終的に有効な`window.priceFromStock`（L8686付近）は既に`seasonPriceDelta`のみを使う実装で、在庫比率（`priceMultMin`/`priceMultMax`）を使う旧実装（L5415, L8392）は後続の代入で上書きされ**実行時には呼ばれない**。`recomputePrices`も同様に最終版（L8707、L10627経由）だけが有効。
- つまり静的読解では「排除すべき値は既にコード上参照されていない」。実行者は以下の手順で実機確認してから対応すること:
  1. 新規ゲームで特定品目（例:鉄）の在庫を大きく変化させても（大量購入/売却）、同一季節内で買値が変わらないことを確認する（変わったら未発見の経路がある）。
  2. 変わらないことが確認できたら、**#16は対応不要**として`docs/修正表.md`に「調査の結果、対象コードは既に排除済みで実害なし」と追記し、状態を「完了」にする。
  3. もし実機で在庫比率的な価格変動が確認できた場合は、`grep -n "priceMultMin\|priceMultMax"`で全参照箇所を洗い出し、実際にどの経路が呼ばれているか特定してから、その経路をseasonPriceDeltaのみの計算に修正する。
  4. なお「季節ごとに店の**在庫数**が増減する」`seasonStockDelta`は価格とは別の意図した仕様なので触らないこと（ユーザーが価格変動と誤認している可能性があるため、実機確認時にどちらの変化かを区別すること）。
- 併せて、無効化されている旧実装（L5415, L8392, および`recomputePrices`のL5440/L6283/L6934/L8420）は死んだコードとして残っている。実害はないが紛らわしいため、時間があれば削除してもよい（必須ではない）。
