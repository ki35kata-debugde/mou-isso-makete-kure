## 1. 仕様（ToBe）と現状（AsIs）

### ToBe（正しい流れ）

| 順 | 条件 | プレイヤー体験 |
|----|------|----------------|
| 1 | 兵糧太が**戦地壱をクリア**（供給を受け**完納**） | ノード完遂・進軍 |
| 2 | **結果フェーズ** | 「人足増員」進言の**解禁通知**のみ（上限人数はまだ増えない） |
| 3 | **次ターン・手紙フェーズ** | 進言に「人足増員」が選択可能になる |
| 4 | 進言クリック（150両） | そのとき初めて `portersMax` +4（4→8→12→16） |

設計上の状態:

- `porterLvMax` … 解禁ストック（`min(3, clearedMax('food'))`）。突破だけで上がる
- `porterLv` … **進言を適用した回数のみ**増える（0〜3）
- `portersMax = 4 + porterLv × 4` … 表示・空き計算の上限

### AsIs（ユーザー実機）

- 戦地壱は**未クリア**（完納していない）
- 前フェーズで依頼条件を満たす**輸送を出発**
- **1ターン後（荷はまだ輸送中）の輸送フェーズ**で `人足(4/8)` を確認
- 結果の解禁告知・進言クリックのいずれも経ていない

解釈: 分母 8 ＝ `portersMax===8` ＝ `porterLv===1` が成立している可能性が高い。  
分子 4 ＝ 空き4。人足4人を出したままなら busy=4・max=8 → free=4 と一致し、「上限だけ不正に+4された」説明と矛盾しない。

## 2. コード調査結果（静的）

### 2.1 上限の算出箇所（正の導出は1系統）

`refreshFacilityUnlockCaps`（閉包約5438–5444行 / window約5994–6000行）:

```js
gs.facility.porterLvMax = Math.min(3, clearedMax('food'));
if ((gs.facility.porterLv||0) > (gs.facility.porterLvMax||0))
  gs.facility.porterLv = gs.facility.porterLvMax;
gs.portersMax = 4 + (gs.facility.porterLv||0)*4;
```

- **上限を+4するのは `porterLv` 増加時のみ**（表示はすべて `gs.portersMax` 参照）
- `porterLv` を +1 するのは `applyPendingAdvice` の `v==='porter_up'` 分岐のみ（約5829・6087行）
- 突破だけでは `porterLvMax` だけが上がる想定

### 2.2 クランプの意味（#26再現条件への含意）

`porterLv > porterLvMax` のとき `porterLv` を **下げて**揃える。

したがって **`portersMax===8` が安定して出る**なら、通常は次が同時成立している:

1. `porterLv === 1`
2. `porterLvMax >= 1`（でなければ次の `refreshFacilityUnlockCaps` で `porterLv` が 0 に戻る）
3. ⇒ `clearedMax('food') >= 1` ⇒ `gs.cleared.food` に 1 以上が入っている

ユーザー体感「戦地壱未クリア」と矛盾する場合の第一候補:

- **内部では完納扱い（`cleared` / `delivered`）が進んでいるが、地図・UIはまだ輸送中**（#3系の到着・完納タイミングと表示のズレ）
- または **一度 `porterLv=1` / `portersMax=8` になったあと、`refreshFacilityUnlockCaps` が走らない経路で表示だけ残る**（可能性は低め・要実機）

### 2.3 進言適用の多重ラップ（#23導入後の構造リスク）

| 層 | 場所 | 挙動 |
|----|------|------|
| ローカル `applyPendingAdvice` | 約5806 | `porter_up` で `porterLv+1`。**shingen を消費しない** |
| `window.applyPendingAdvice` | 約6063 | 同上＋他進言 |
| 二重適用ガード | 約7512 | `appliedThisTurn` なら return |
| 即時適用 | 約7483 `applyAdviceNow` | 課金成功後に window 版を呼ぶ |
| `goToNextTurn` 先頭 | 約5840 | **裸の `applyPendingAdvice()`（ローカル）**を呼ぶ |
| ターン跨ぎ消費 | 約7528 | `appliedThisTurn` なら `shingen=null` してから orig |

リスク:

- ローカル版と window 版の**二重定義**で、ガードをすり抜けた経路があると `porterLv` が意図せず増える
- `shingen==='porter_up'` が**プレイヤー操作以外**で残っていると、ターン送りや即時適用で発動する
- `porter_up` のロック解除（`porterLvMax > porterLv`）と課金ラップの順序で、**ロック前に値が入る**経路がないか要確認

### 2.4 解禁通知（上限は増やさない）

`refreshFacilityUnlockCaps` の post-hook（約11350行）は `porterLvMax` 増加時に今季ログへ告知するだけ。**`porterLv` は触らない**。  
ここは ToBe の「結果で解禁通知」に相当し、#26本体（上限増加）の直接原因にはならない。

### 2.5 表示

輸送UIは一貫して `空き{portersFree()}/{gs.portersMax||4}`。  
分母を `porterLvMax` と取り違えるコードは見当たらない → **表示バグより状態バグ優先で追う**。

## 3. 想定原因（優先順）

| 優先 | 仮説 | 裏付け | 切り分け |
|------|------|--------|----------|
| A | **`porter_up` が進言未選択でも適用された**（残留 `gs.kengen.shingen`、二重 `applyPendingAdvice`、課金ラップの誤発火） | +1 経路が実質ここだけ | 再現直後に `gs.kengen` / `porterLv` / トースト有無を確認 |
| B | **内部では N1 完納済み**（`cleared.food` に1）だが UI は輸送中のまま。そのうえで何らかの経路で進言相当が走った、または旧セーブの `porterLv` | クランプで max=8 を維持するには `porterLvMax≥1` が必要 | `gs.cleared` / `gs.order.food` / `convoys` / 地図駒を同時確認 |
| C | **#3 到着処理の移動**により、出発翌ターンの結果／輸送再描画タイミングで完納・解禁が先走る | ユーザー「条件を満たす輸送の1T後・輸送中」と時間軸が一致しうる | `departTurn`・`eta`・`delivered`・`tickConvoys` 呼び出し位置をトレース |
| D | セーブ／途中再開で `portersMax` または `porterLv` が汚染 | 新規ゲームで再現しなければセーブ由来 | 新規ゲーム限定で再現テスト |

### 4.0 実機確認手順（必須・5分）

再現直後、コンソールで:

```js
({
  turn: gs.turn,
  portersMax: gs.portersMax,
  porterLv: gs.facility && gs.facility.porterLv,
  porterLvMax: gs.facility && gs.facility.porterLvMax,
  clearedFood: gs.cleared && gs.cleared.food,
  nodeFood: gs.node && gs.node.food,
  orderFood: gs.order && gs.order.food,
  kengen: gs.kengen && {
    shingen: gs.kengen.shingen,
    paid: gs.kengen.paidThisTurn,
    applied: gs.kengen.appliedThisTurn
  },
  convoys: (gs.convoys||[]).map(function(c){
    return {route:c.route, eta:c.eta, departTurn:c.departTurn, porters:c.porters, cargo:c.cargo};
  }),
  busy: typeof portersBusy==='function' ? portersBusy() : null,
  free: typeof portersFree==='function' ? portersFree() : null
})
```

判定:

| 観測 | 次にやること |
|------|----------------|
| `porterLv===1` かつ `clearedFood` 空 | 仮説A（進言誤適用）＋クランプ未実行。適用経路を特定 |
| `porterLv===1` かつ `clearedFood` に1 | 仮説B/C。完納タイミングと UI のズレを直し、あわせて誤適用の有無を見る |
| `porterLv===0` なのに表示8 | 表示／別 `portersMax` 汚染。描画関数を特定 |
| `kengen.shingen==='porter_up'` なのに未操作 | 設定経路を特定 |

### 4.1 修正F1 — 進言適用の単一路化（ほぼ必須）

1. **`porterLv` を増やすコードパスを1つに統一**（`window.applyPendingAdvice` のみ）
2. ローカル `applyPendingAdvice`（約5806）は window へ委譲するか削除
3. `goToNextTurn` 先頭の裸呼び出しは **window 版＋「未適用のときだけ」** に限定
4. 適用成功後は必ず:
   - `gs.kengen.shingen = null`（または消費フラグでガード）
   - `appliedThisTurn` / `_appliedValues` を一貫管理
5. `porter_up` 適用条件を厳格化:

```js
// 擬似コード
if (v === 'porter_up') {
  if ((gs.facility.porterLvMax||0) <= (gs.facility.porterLv||0)) return; // 解禁枠なし
  // 念のため: 突破実績が無いなら適用しない（LvMax と二重でも可）
  if ((clearedMax('food')||0) < 1 && (gs.facility.porterLvMax||0) < 1) return;
  gs.facility.porterLv = (gs.facility.porterLv||0) + 1;
  refreshFacilityUnlockCaps();
}
```

### 4.2 修正F2 — 解禁と効果の責務分離を明文化（コードコメント＋ガード）

- **解禁** = `porterLvMax` のみ（完納時の `refreshFacilityUnlockCaps`）
- **効果** = プレイヤーが `porter_up` を選び課金したときだけ `porterLv++`
- 完納処理（`checkOrderComplete` / `markNodeCleared` / ノード前進）から **`porterLv` を絶対に触らない**

### 4.3 修正F3 — 完納タイミングと「輸送中」表示の整合（仮説B/Cが真のとき）

#3 対応（結果フェーズ前の `tickConvoys`）と干渉していないか確認:

1. 出発ターンは ETA を減らさない（既存 `departTurn` ガード）
2. **完納・`cleared`・駒移動は到着確定時のみ**
3. 輸送中コマがある間は、依頼が画面上「未完納」と内部 `delivered` が矛盾しないこと
4. もし「依頼数量ちょうど出発した時点」で `delivered` を先に足している経路があれば削除

※ F3 は #3 と同一コードを触るため、直す場合は **#3 の受け入れ条件を再検証**すること。

### 4.4 修正F4 — セーフティ（セーブ汚染対策）

`refreshFacilityUnlockCaps` 内で毎回:

```js
// porterLv は Max を超えない（既存）
// 追加: portersMax は常に式から再計算（既存どおり）
// 任意: porterLv が数値でない・負なら 0 に正規化
gs.facility.porterLv = Math.max(0, Math.min(
  gs.facility.porterLvMax||0,
  Number(gs.facility.porterLv)||0
));
gs.portersMax = 4 + (gs.facility.porterLv||0)*4;
```

※ 既存クランプは「超過時に Max へ」で、負や NaN は未処理。正規化を明示するとセーブ汚染に強い。

**注意:** この正規化だけでは「未クリアなのに Lv=1」は直せない（Max≥1 のとき）。**根本は F1＋完納条件**。
