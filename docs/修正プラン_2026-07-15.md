## #2 市場の前期比がT2以降ずっと±0（バグ・原因特定済み）★最優先・最小修正

### 原因
ターン送り時の実行順序の問題。
1. `goToNextTurn` の外側ラッパー（L8481付近「⑤ ターン進行: 前期保存 → 季節在庫 → 今季価格」、および L6892付近）が `gs.market.prevBuy` に**旧季節の価格**を正しく保存する。
2. コア `goToNextTurn`（L2010）が `gs.season` を進めた後、L2028 の `updateInvDisplay()` を呼ぶ。`updateInvDisplay` はラッパー（L5805付近）経由で `refreshMarketUI()` → `recomputePrices(false)` を呼び、この時点で `priceBuy` が**新季節価格**になる。
3. その後、L5779（「goToNextTurn: season market + advice apply」IIFE内）の `recomputePrices(true)` が `prevBuy = priceBuy`（＝すでに新季節価格）で**上書き**してしまう。
4. 結果 `prevBuy === priceBuy` となり、`formatDelta` が常に（±0）を返す。

### 修正（1文字レベルの最小修正）
L5779 付近:
```js
      seasonRestock();
      recomputePrices(true);   // ← これを
      recomputePrices(false);  // ← こう変える
```
`prevBuy` の保存は L8481 / L6892 の**ターン送り前**スナップショットに任せる（これらは正しい値を保存している）。

### 検証
1. 新規ゲーム開始（古いセーブは使わない）→ T1 経営フェーズで市場を開き価格をメモ
2. ターンを送って T2（夏）で市場を開く
3. `data/market_economy.json` の `seasonPriceDelta` どおりの差が表示されること（例: 鉄 春+2→夏-2 なので「(-4)」、全品目±0でないこと）
4. T3（秋）でも同様に変動が出ること
5. 売買を確定しても前期比が変わらないこと（`confirmTrade` ラッパー L6879 が prevBuy を保護している）

## #3 進言費用：褒める=300両／それ以外=150両（バグに近い仕様不整合）

### 原因
費用の**表示**は末尾パッチ（L10789 `window.adviceCostFor`）で「commend=300／その他=150」に対応済みだが、**実際の課金**は古いIIFE内のローカル関数を使っていて常に300両になる：

- L6196 `function adviceCost()`（「advice+sell」IIFE内）: `MARKET_ECON.adviceCostGold`（=300）を無条件に返す。同IIFEの `kgPick` ラッパー（L6409）が L6413 `var cost = adviceCost();` で課金判定に使用。
- L6676 `function adviceCost()`（「UI polish kengen/market/ranch」IIFE内）: 同じく300固定。同IIFEの `kgPick`（L6767）で使用。
- 実際の課金はチェーン最深部の L6424（または L6780）で `gs.gold -= cost` されるため、**全進言が300両**になる。
- 末尾の `window.adviceCost` 上書き（L10800）はIIFE内のローカル関数を差し替えられない（スコープが違う）ため効かない。

### 修正（2箇所のローカル関数を val 対応にする）
L6196 と L6676 の2つの `function adviceCost()` を、それぞれ以下に置き換える（同名・同シグネチャ拡張なので呼び出し側の互換は保たれる）:

```js
function adviceCost(val){
  // 末尾パッチの共通ロジックがあればそれを使う（commend=300/その他=150）
  if(typeof window.adviceCostFor==='function'){
    var v = (val!==undefined) ? val
      : (window.__adviceCostPendingVal!=null ? window.__adviceCostPendingVal
        : (gs.kengen && gs.kengen.shingen));
    return window.adviceCostFor(v);
  }
  var c = window.MARKET_ECON && window.MARKET_ECON.adviceCostGold;
  return (c!=null ? c : 300)|0;
}
```

さらに呼び出し側に val を渡す（両IIFEの kgPick 内）:
- L6413: `var cost = adviceCost();` → `var cost = adviceCost(val);`
- L6767: `var cost = adviceCost();` → `var cost = adviceCost(val);`

（L6404・L6702 の表示系 `adviceCost()` は引数なしのままで良い。表示は L10820 の `paintKengenCostsAndLocks` ラッパーがボタンごとに `adviceCostFor(val)` で上書きするため。）

注意: `window.adviceCostFor` は L10789（ファイル末尾）で定義されるが、課金はユーザー操作時（実行時）なので参照可能。起動時の `console.log('[advice+sell] ready cost='+adviceCost())`（L6533）はフォールバック側を通るだけで無害。

### 検証
1. 新規ゲーム → 手紙フェーズ → 謁見タブ
2. 「王子を褒める」以外の進言（鉄増産など）を選択 → **金が150両だけ減る**こと（現状は300減る）
3. 「王子を褒める」を選択 → 300両減ること
4. 各ボタンの費用表示（費用：金150両／金300両）と実際の課金が一致すること
5. 所持金140両で150両の進言を選ぶと拒否されること（残金チェックも同じcostを使うため自動で整合するはず）

## #1 輸送フェーズで「現庫0」と表示されることがある（バグ）

### 症状
剣の在庫が3あるはずなのに出立準備（手渡し）欄が「現庫: 0」。＋を押すと表示が実在庫ベースに再計算される（＝クリックで初めて直る）。

### 原因（推定・高確度）
- 静的HTMLの初期値が `現庫: <b>0</b>`（L877/905/916/945/988）。
- 正しい値への更新は `updateTransportUI` 内（L2154）と `changePrepQty`（L1933→L1942）でしか行われない。
- `setPhase` は12回以上ラップされており（L1641 のコアから L10735 まで）、輸送フェーズ突入時に `updateTransportUI` が呼ばれても、途中のラッパー（L3719 など）のフェーズガードが `gs.phase` 設定タイミングと競合して実質更新されないケースがある。馬王子の欄だけは L4520 のラッパーが `p==='transport'` 時に setTimeout で強制更新しており（L4527-4530）、**他の王子には同等の処理がない**。
- なお「3あるはずが2」の差は、＋押下時の再計算が正しい在庫（`gs.stock.sword`）を表示した結果。生産中（`gs.inProd`）は現庫に含まない仕様のため、ユーザーの想定在庫とずれる可能性もある（仕様通り）。

### 修正（末尾追記パッチ・既存行は触らない）
ファイル末尾（最後の `</script>` の直前ではなく、**新しい `<script>` ブロックとして追加**）に以下を追記する:

