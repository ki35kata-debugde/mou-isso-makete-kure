## #7 進言費用：「褒める」が150両になっている（本来300両）★最優先・原因特定

### 原因
前回修正で `adviceCost(val)` は正しく `val==='commend'` のとき300を返すようになった（L10789 `adviceCostFor`、L6196 `adviceCost`）。しかし **褒めるボタンの実際のクリック時に渡る `val` は文字列 `'commend'` である**（L1277 `onclick="kgPick('shingen','commend',this)"`）ため、一見问题ないはずだが、実際には150両で課金されている。

調査の結果、`kgPick` は複数のIIFEで多重ラップされている。課金を実行するチェーンの**最も外側**は L6414-6439（「進言コスト」IIFE）で、ここは `adviceCost(val)` を正しく呼んでいる（修正済み）。しかしこれより**後に定義される**別のラッパー（L6767付近「UI polish kengen/market/ranch」IIFE内の `kgPick`、および場合によっては他の同名ラッパー）が存在し、JSでは同名関数の再代入は**後勝ち**になるため、実行時に有効なのはファイル内で最後に代入された `kgPick` 関数である。

L6767のIIFE内には旧来のローカル `adviceCost()`（引数なし、L6682で定義）を呼ぶ課金コードが残っている可能性が高い（前回プランの#3adviceCost修正で「6196、6682」両方を修正対象としたが、**課金呼び出し側の書き換えがL6767のIIFE内でも行われているか要再確認**）。もし6767側の `var cost = adviceCost();`（引数なし）のままであれば、L6682のadviceCost(val)は `val=undefined` で呼ばれ、`adviceCostFor(undefined)` が実行される。`adviceCostFor` の実装（L10789）は:
```js
window.adviceCostFor = function(val){
  if(val==='commend'){ ... return 300; }
  ... return 150; // val!=='commend' なので undefined でも150になる
};
```
→ **これが原因**。`val` が渡っていないと必ず150両になる。

### 修正
1. L6767付近（またはそれ以降に定義される最終的な `kgPick`）を検索し、`var cost = adviceCost();`（引数なし呼び出し）を見つけたら、その直前の `type,val,btn` の `val` を使って **`var cost = adviceCost(val);`** に変更する。
2. `grep -n "window.kgPick = function" mou_isso_v0_6_1.html` で全ての `kgPick` 再代入箇所を洗い出し、**ファイル内で最後に定義されているもの**を特定し、そこが `adviceCost(val)` を呼んでいることを保証する。もし複数箇所に課金ロジックがあり、どれが最終有効かの判断が難しい場合は、確実な方法として **`window.adviceCostFor` を直接使うよう統一**する：
   ```js
   var cost = (typeof window.adviceCostFor==='function') ? window.adviceCostFor(val) : adviceCost(val);
   ```
   をすべての課金箇所（`gs.gold -= cost` の直前で cost を計算している全箇所）に適用する。

### 検証
1. `grep -n "kgPick = function" mou_isso_v0_6_1.html` で行番号一覧を取得し、最後の行番号のブロックを読んで cost 計算ロジックを確認
2. 新規ゲームで「王子を褒める」進言を選択 → 金が**300両**減ること
3. 他の進言（鉄増産など）は引き続き150両であること（後退確認）

## #6 荷車の表記・仕様変更＋解禁ポップアップ

### 現状
- 荷車ボタンの表示は「荷車（空き{freeC}/{maxC}・積載2）」（L9557、および L3081/8881等の類似箇所は人足向けで別ロジック）。空き人数ベースの表示だが、実際は「金20両かかるが積載2で運べる」仕様にしたい。
- 解禁通知：`queueUnlock('unlock_cart', '荷車が解禁されました。...')`（L3779）で通知テキストは存在する。`queueUnlock`が実際にOKボタン付きポップアップとして表示されているか、それとも別のトースト等で目立たず流れているかを確認する必要がある。

### 調査すべき点（実行者が最初に行うこと）
1. `grep -n "queueUnlock\|flushGamePopup\|_pendingGamePopup" mou_isso_v0_6_1.html` でポップアップ機構の全体像を掴む。既存の牧場仔馬ポップアップ（L8630付近 `gs._pendingGamePopup = {title:..., body:...}` → `flushGamePopup()`）と同じパターンを踏襲する。
2. `queueUnlock('unlock_cart', ...)` が実際に画面上でOKボタン付きモーダルとして出るコードパスを確認する（出ていない、もしくは弱い通知なら差し替える）。

### 修正方針
1. **表示変更**: 荷車ボタンのラベル生成箇所（`t:'荷車（空き'+freeC+'/'+maxC+'・積載2）'` のパターン。該当箇所を `grep -n "荷車（空き" mou_isso_v0_6_1.html` で全て洗い出し、見つかった全箇所を `'荷車（要20金・積載2）'` に統一する。
2. **課金の実装**: 荷車を選択して出発する処理（`goTransport`のcart分岐、または `selectMeans('cart',...)` 実行時）に、出発時点で `gs.gold -= 20`（残金不足なら出発不可＋トースト）を追加する。既存の人足・馬車と処理を揃えるため、`goTransport`ラッパー群（L9032, L9647等）のうち最終的に有効なものに手を入れる。末尾追記パッチとして以下のような形が安全:
   ```js
   var __goTransportCartFee = window.goTransport;
   window.goTransport = function(prince){
     try{
       var t = gs.transport && gs.transport[prince];
       if(t && t.means==='cart'){
         if((gs.gold||0) < 20){ showToast('荷車の利用には金20両が必要です'); return; }
         gs.gold -= 20;
         if(typeof updateHeaderDisplay==='function') updateHeaderDisplay();
       }
     }catch(e){ console.warn('cart fee', e); }
     return __goTransportCartFee.apply(this, arguments);
   };
   ```
   （実行者は既存の `gs.transport[prince].means` の実データ構造をgrepで確認してから当てはめること。）
3. **解禁ポップアップ**: `unlock_cart` の解禁トリガー箇所（L3779付近 `if(key==='food'&&node===1&&!beforeFood1) queueUnlock('unlock_cart', ...)`）で、`queueUnlock` の実装が弱い通知（トースト等）である場合、牧場ポップアップと同じ `gs._pendingGamePopup = {title:'輸送手段の解禁', body:'荷車が使えるようになりました。\n利用には金20両かかりますが、積載2で運べます。'}` → `setTimeout(function(){ window.flushGamePopup(); }, 450);` のパターンに差し替える。既存の `queueUnlock` 機構がすでにOKボタン付きモーダルなら、本文だけ「金20両かかりますが積載2で運べます」を追記する。

### 検証
1. 新規ゲームでノード1撃破前後の荷車ボタン表示を確認 → 「荷車（要20金・積載2）」になっていること
2. 荷車で輸送を出発 → 所持金が20両減ること、残金19両以下なら出発できずトーストが出ること
3. 解禁のタイミングでOKボタン付きポップアップが表示され、内容が「金20両かかるが積載2で運べる」旨になっていること

## #5 馬の王子：出立直後に同ターン内で納品できてしまう

### 現状
- 他の王子は `canTransportThisTurn(key)`（L3068）で `gs.sortieTurn[key]` と `gs.turn` を比較し、出立した同ターンの輸送をブロックしている（L3072 `if(gs.sortieTurn[key]!=null && gs.turn<=gs.sortieTurn[key])return false;`）。
- 馬の王子は輸送手段選択UIを使わず、専用の「馬を選んで送る」UI（`horse-send-box`、L8940付近）と `goTransportHorsePrince()`（L9103）を使う別経路になっている。この経路には `canTransportThisTurn` のチェックが入っていない可能性が高い。

### 調査すべき点
1. `goTransportHorsePrince`（L9103付近）の中身を読み、`canTransportThisTurn('horse')` の呼び出しがあるか確認。
2. `renderHorseSendBox`相当の関数（L8940のUIを生成している関数。おそらく `L8824`前後の「buildMeans horse分岐」やそれを呼ぶ親関数）が、出立直後のターンでもUIを表示してしまっていないか確認。

### 修正方針
1. `goTransportHorsePrince()` の冒頭に、他の王子と同様のガードを追加:
   ```js
   if(typeof canTransportThisTurn==='function' && !canTransportThisTurn('horse')){
     showToast('出立したターンは輸送できません（次ターン以降）');
     return;
   }
   ```
2. 馬選択UI自体（horse-send-box）も、出立直後のターンは表示しない・または操作不可にする。UIを生成している関数の冒頭に:
   ```js
   if(gs.sortieTurn && gs.sortieTurn.horse!=null && gs.turn<=gs.sortieTurn.horse){
     box.innerHTML = '<div style="color:var(--faded)">出立した季節は輸送できません。次のターンからお願いします。</div>';
     box.style.display = 'block';
     return; // Go ボタンも生成しない
   }
   ```
   実行者は該当関数の実際の変数名・returnの型を確認してから当てはめること。

### 検証
1. 新規ゲームで馬の王子を出立させる（ノード0→1）
2. 同じターン内で輸送フェーズの馬王子欄を見る → 馬選択チェックボックスが出ない、または「次のターンから」の案内が出て送れないこと
3. 次のターンに進めると通常通り馬を選んで送れること

## #4 ノード2の武器/兵器の皇子への輸送で品目行が小さく暗くなる

### 現状（推定）
スクリーンショットでは剣の品目行（アイコン・現庫4・依頼残4・輸送中0・−/+ボタン）全体が薄いグレーで表示され、直前の行（荷車ボタン）はロック中スタイル（`opacity`低め）になっている。品目行自体に`opacity`を下げるクラスが誤って付与されている可能性がある。

### 調査すべき点
1. 該当ノード（weapon/siegeのノード2到達時）に特有の分岐がないか `grep -n "node===2\|node:2\|node_gte.*2" mou_isso_v0_6_1.html` で確認。
2. `.tp-item-line`（L677定義）に対して動的に別クラス（`locked`等）が付与されるコードがないか、`ensureOrderUI`のレンダラ（L9463〜9490、L3688〜3714）を再確認。
3. アイコン注入ラッパー（L10613〜）が親要素にstyleを当てていないか確認。

### 修正方針（調査結果次第で分岐）
- もし特定のクラス付与が原因なら、そのクラス付与条件を修正（誤爆している条件式を直す）。
- 原因がCSSの継承（親のロック済み要素からopacityを引き継いでいる）であれば、`.tp-item-line`に明示的に `opacity:1` を指定するCSSを追記する。
- 原因が特定できない場合は、`.tp-item-line`要素とその親のクラス一覧をconsole.logで出力するデバッグコードを一時追加して実行者が原因を絞り込んでから修正すること（このケースは静的検証だけで断定しにくいため、実行者の判断で追加調査してよい）。

### 検証
1. 兵糧太で兵糧を送りノード1に到達させた後の輸送画面と見比べ、武器/兵器の皇子がノード2到達時の輸送画面で品目行の見た目（文字色・不透明度）が同じになること
2. スクリーンショットの状態（暗く・小さく見える）が再現しないこと
