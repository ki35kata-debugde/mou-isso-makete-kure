# 修正プラン 2026-07-16 — 進言まわり一括対応（#4・#13・#23）

対象: [修正表.md](./修正表.md) #4（旧#30）＝バグ本体＋市場強化再設計、#13（旧#37）＝増産系進言の削除、#23＝進言「人足を増やす」の新設。
いずれも同じ進言コード群（謁見ボタンHTML・`kgPick`・`applyPendingAdvice`・`refreshFacilityUnlockCaps`・ロック描画）を触るため1バッチで対応する。

## 1. 調査結果（ブラウザ実機で再現・確認済み）

ローカルサーバー（`npx http-server -p 8642`、`.claude/launch.json`に`game`として登録済み）で
`mou_isso_v0_6_1.html` を起動し、兵糧太N1の依頼を輸送到着→完納させる実プレイ相当の手順で再現した。

### 再現結果

完納ターン終了後の状態:

| 項目 | 値 | 判定 |
|------|-----|------|
| `gs.cleared.food` | `[1]` | ✅ 突破は記録されている |
| `gs.facility.marketLvMax` | `1` | ✅ 内部状態は解禁済み |
| `#kg-market-up` ボタン | `disabled=true`・`locked-opt`クラス残存 | ❌ **UIだけ鎖されたまま** |

つまり**ゲーム状態は正しく解禁されているのに、ボタンのロック表示だけが永遠に解除されない**。
クリックは`disabled`のため発火せず、「選択できない」の報告と完全に一致する。

### 根本原因

進言ボタンのロック解除は2系統あるが、噛み合っていない:

1. **`refreshAdviceButtons`（新版・`window`側 約6058行）**
   `locked-opt`クラスと`disabled`を状態から再計算する唯一の関数。
   ただし実ターンフローの再描画チェーン（`refreshMarketUI`内部 約5535行の裸呼び出し）では
   **旧版（クロージャ側 約5540行）が解決されて呼ばれる**ため、`locked-opt`クラスが残る。
2. **`paintKengenCostsAndLocks`（約6741行）**
   謁見タブを開くたび・ターン送り後に呼ばれるが、約6772行で
   `btn.classList.contains('locked-opt')` という**古いクラスを入力として信頼**し、
   `disabled=true` を復元してしまう。

検証: `window.refreshMarketUI()`を呼んでもロックは解除されない（旧版経由＋直後のpaintで再ロック）。
`window.refreshAdviceButtons()`を直接呼ぶと即座に解錠され、その後の`kgPick('shingen','market_up')`は
課金・適用まで正常動作する（実測: 150両課金、marketLv 0→1、shopCap 20→28）。

### 副産物の発見（今回は対象外、修正表#3・#20に記録）

`window.tickConvoys`のラップ群（ノード前進処理 約10250行、馬deployed化 約9214行）は
**実ターンフローから一切呼ばれていない**（実測: ログフック無反応のまま到着処理が完了する）。
本体の`goToNextTurn`ラップ(約3585行)内の裸`tickConvoys()`呼び出しが、windowラップではなく
元の実装に解決されるため。#3（1ターン遅延）と#20（馬輸出の巻き戻り）の根本原因に直結する。
また約4376行・約5065行に「毎ターン全`transit`馬を`ranch`へ強制返却」する処理があり、これも#20の直接原因。

## 2. 修正方針

### 修正A（バグ本体）: `paintKengenCostsAndLocks`を自己完結型にする

古いクラスを入力にせず、**毎回ゲーム状態からロックを再計算して出力する**よう変更する。

- ループ前に `refreshFacilityUnlockCaps()` を1回実行
- `data-val`ごとの解禁判定表を持つ:
  - `staff_smith`: `smithStaffMax > smithStaff`
  - `staff_pharm`: `pharmStaffMax > pharmStaff`
  - `market_up`: `marketLvMax > marketLv`
  - `stable_up`: `stableExtraMax > stableExtra`
- 約6766〜6777行の分岐を「解禁判定 → `locked-opt`クラスと`disabled`を設定」に置換
  （`kg-locked-other`＝選択済みロックの動作は現状維持）

該当コードはHTML内に1箇所のみ。どの`refreshAdviceButtons`が呼ばれても結果が正しくなる。

### 修正B（仕様拡張②）: 市場強化の再設計

現状: `marketLvMax = staffUnlockMax('food')`（最大2）、`shopCap = 初期値 + marketLv×8`、入荷量は無関係。

変更（**クロージャ版とwindow版の両方を同一内容で書き換える**こと）:

| 項目 | 変更後 | 該当箇所 |
|------|--------|----------|
| 解禁数 | `marketLvMax = min(4, clearedMax('food'))`（戦地1〜4突破ごとに+1、最大4ストック） | 約5413-5414行（閉包版）・約5950行（window版） |
| 店在庫上限 | `shopCap[k] = min(99, 初期値 × (1 + marketLv))`（Lv1〜4で2/3/4/5倍） | 約5424行（閉包版）・約5960行（window版） |
| 入荷量 | 季節テーブルの**正の加算量**を `×(1 + marketLv)`（負値＝季節消費はそのまま）。クランプは既存のcap/min処理に任せる | `applySeasonStockDelta` 約8479-8488行（window版）・`seasonRestock` 約5483-5486行（閉包版） |
| 告知 | 既存の解放検知フック（約11289行、今季ログに赤字告知）をそのまま活用 | 変更なし・動作確認のみ |

補足: ambient（全品目+1の自然入荷）は増幅しない（判断が分かれる場合はユーザー確認）。

### 修正C（仕様拡張①）: 工房・薬房増員の生産枠+3

**実装済みを確認**（`smithCap()=3+smithStaff*3`・`pharmCap()`同様、約5432-5433行。適用時トースト表示も+3表記）。
コード変更不要。実機確認のみ。

### 修正D（#13）: 「XX増産」系進言の削除

削除対象は**鉄増産・木材増産・兵糧増産・薬草増産の4種**（名馬探索は残す）。

- HTML 約1252〜1267行の4ボタン（`data-val="iron"/"wood"/"food_mat"/"herb"`）を削除
- `window.applyPendingAdvice`（約6018行）内の増産分岐（約6023-6025行、`applyBoostShop`呼び出し）と
  `applyBoostShop`／`yearRestockAmount`本体（約5984-6014行）を削除
- `SHINGEN_LABELS`（約5726行）と`kgPick`内ラベル表（約6131-6138行）から該当4キーを削除
- 閉包版`applyPendingAdvice`（約5769行）には増産分岐がもともと無いため変更不要

### 修正E（#23）: 進言「人足を増やす」の新設

**仕様**: 兵糧太ルートの戦地壱・弐・参突破ごとに進言枠+1（最大3ストック）。1回選ぶごとに人足上限+4
（`gs.portersMax`: 4→8→12→16）。費用は「その他」進言と同じ150両（`adviceCostFor`のデフォルトのまま）。

**状態設計**: `gs.facility.porterLv`（適用済み回数 0〜3）・`gs.facility.porterLvMax = min(3, clearedMax('food'))`。
`gs.portersMax = 4 + porterLv*4` を `refreshFacilityUnlockCaps` 内で毎回導出する（既存セーブも自動移行、
他の全箇所は `gs.portersMax||4` 参照なので影響なし）。

**変更箇所（既存の`staff_smith`/`market_up`と同じパターンを踏襲）**:

| 箇所 | 内容 |
|------|------|
| HTML `#kg-shingen`（約1284行の厩舎拡張の後） | 新ボタン `id="kg-porter-up"` `data-val="porter_up"`、タイトル「人足を増やす」、効果「人足の上限+4　条件：兵糧太が戦線を突破時に解放」 |
| `refreshFacilityUnlockCaps` **両方**（閉包版約5398行・window版約5934行） | `porterLvMax`計算・`porterLv`クランプ・`gs.portersMax`導出を追加 |
| `kgPick`実効版（約6088行の置換版） | `porter_up`の未解放/上限ガード＋トースト追加。ラベル表（約6131行）にも追加 |
| `window.applyPendingAdvice`（約6018行）と閉包版（約5769行） | `porter_up`分岐: `porterLv+1`→`refreshFacilityUnlockCaps()`→トースト「人足が4人増えました（上限X人）」 |
| `refreshAdviceButtons` window版（約6058行）の`setLock`群 | `kg-porter-up`のロック判定追加（`porterLvMax > porterLv`） |
| 修正Aで自己完結化する`paintKengenCostsAndLocks`の判定表 | `porter_up`エントリ追加 |
| 解放告知フック（約11296行`labels916`と約11138行スナップショット） | `porterLvMax:['兵糧太','人足を増やす']`とスナップショットキー追加 |
| 閉包版`refreshAdviceButtons`（約5540行）・閉包版`kgPick`ラップ（約5732行） | 実フローでの実効性が不確かなため（二重スコープ問題）、**同内容を追加しておく**（害はなく、防御的に一致させる） |

**備考（ストックの挙動・ユーザー確認済み 2026-07-16）**:
- 兵糧太突破は市場強化枠（修正B）と人足枠（修正E）の**両方を同時に**解禁する
- 解禁された枠は**使用期限のないストック**。今季使う必要はなく、何ターン後でも謁見画面から使える
- 「1ターン1回」はそのターンに選べる進言が1つという制約であり、ストックは消滅しない
- 残ストック（Max−使用済み回数）が1以上ある限りボタンは選択可能のまま。0になった時のみロック表示
- 実装上は `marketLvMax`/`porterLvMax`＝突破数（解禁総数）、`marketLv`/`porterLv`＝使用済み回数、が
  そのままストック台帳になるため、追加の期限管理・ターン管理は不要

## 3. 実装形式

引継ぎガイドの規約に従い、新規 `data/fix_kengen_overhaul.js`（Node実行でHTML本体を文字列置換、
実行前にバックアップ作成。修正A/B/D/Eを1スクリプトで適用）。マーカー: `[kengen overhaul] ready`。

注意（引継ぎガイド既知負債）: 対象関数はクロージャ版とwindow版の二重定義があるため、
**末尾IIFE追加ではなく該当関数本体の直接置換**が必要。置換パターンが両方に一致することを必ず確認する。

## 4. テスト手順（localhost:8642）

1. `Ctrl+F5` → 新規ゲーム
2. コンソールで兵糧太N1完納を再現:
   `gs.sortied.food=true; gs.node.food=1; setOrderForNode('food',1); gs.convoys.push({route:'food',eta:1,cargo:[{id:'food',qty:gs.order.food.items[0].qty}]}); gs.gold=2000; goToNextTurn();`

4. 選択→150両課金→今季ログに告知→`gs.market.shopCap.iron`が初期値×2（≦99）になること
5. `gs.cleared.food=[1,2,3,4]`にして4回分選択できること・5倍かつ99クランプを確認
6. ターン送り後の入荷量が倍率反映されていること（`gs.market.shop`前後比較）
7. 工房増員選択で生産枠が3→6になること（修正C確認）
8. 増産系4進言（鉄・木材・兵糧・薬草）がボタン一覧から消えていること（修正D確認）
9. 「人足を増やす」: 未突破時はロック表示→`gs.cleared.food=[1]`で解錠→選択で150両課金・`gs.portersMax`が8になり、輸送画面の人足表示が「空き8/8」になること。戦地参まで突破で3回分（上限16）まで積めること、4回目はロックされること（修正E確認）
10. 回帰確認: 厩舎拡張・薬房増員のロック/解錠、進言の選び直し不可、費用表示、名馬探索が残っていること

## 5. リスク・影響範囲

- 修正Aは表示ロジックのみ。状態は変更しないためセーブ互換に影響なし
- 修正BはshopCapが毎ターン`refreshFacilityUnlockCaps`で再計算されるため、既存セーブでも自動的に新式へ移行する
- `marketLv > marketLvMax`時の既存クランプは維持（旧セーブでLv2まで上げていた場合もclearedベースの新Maxで安全に収まる）
