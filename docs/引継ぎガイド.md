## 4. アーキテクチャ（どう動いているか）

### 4.1 単一 HTML に何が入っているか

1. CSS・UI マークアップ  
2. 埋め込み画像（base64）  
3. `GAME_DATA` / `MARKET_ECON` / `UI_STRINGS` 等の JSON ライクな巨大オブジェクト  
4. 本体ロジック（`gs` 状態機械・フェーズ）  
5. 後から足した **IIFE パッチ**（ファイル末尾付近、`console.log('[xxx] ready')` が目印）

### 4.2 状態の中心: `gs`

主なフィールド（抜粋）:

| フィールド | 内容 |
|------------|------|
| `phase` | letter / management / transport / result |
| `turn`, `season`, `year` | 進行 |
| `gold`, `inv`, `stock` | 所持 |
| `trust`, `choices`, `prepDone`, `sortied` | 皇子 |
| `node`, `order`, `cleared` | 前線 |
| `convoys` | 輸送中の荷 |
| `market`, `facility` | 市場・人員 |
| `horses`, `ranch` | 馬・牧場 |
| `mail`, `letterFlags` | 手紙新着・既読 |
| `kengen` | 進言 |
| `portersMax`, `fleet` | 人足・荷車台数 |

### 4.3 フェーズ

```
手紙 → 経営 → 輸送 → 結果 →（次ターン）手紙 → …
```

1 ターン ≒ 1 季節（春0→夏1→秋2→冬3）。

### 4.4 パッチの読み方

末尾に例えば:

```js
// ═══ node advance piece ═══
(function(){
  if(window.__nodeAdvancePiece) return;
  ...
  console.log('[node advance piece] ready');
})();
```

- **後から読まれたものが `window.xxx` を上書き**する  
- 同じ機能が二重にあるとバグる（過去に N2 進行が壊れた原因）  
- 修正時は「最終的にどの `window.xxx` が勝つか」を確認する

## 5. ゲーム仕様の要点（実装準拠）

### 5.1 4皇子

| key | 名前 | 方向 | N0準備 |
|-----|------|------|--------|
| food | 兵糧太 | 東 | 兵糧×5 |
| horse | 騎馬次 | 北 | 兵糧×4・回復薬×1 |
| siege | 攻三城 | 西 | 衝車×2 |
| weapon | 武刃四 | 南 | 剣×2 |

### 5.2 信頼

| タイミング | 値 |
|------------|-----|
| N0 手渡し完了 | +4（`prep_quests.trust_reward`） |
| N1〜N3 完納 | +15 |
| N4 完納 | +20 |
| 娘手紙の選択肢 | 各 choice の `trust`（0〜2 程度） |
| 進言「褒める」 | +10 |

### 5.3 手紙

- 新着: ノード到達（N0〜N4）文官＋娘、信頼 20/40/60/80/100 の娘  
- タブ表示: 状態（出立待ち／戦地壱…）＋ **`[新着！]`（赤）**  
- 未返信で完了: **確認ダイアログ**（はい→進む／キャンセル→留まる）  

### 5.4 進言（謁見）

- 費用 300 両（`adviceCostGold`）  
- 1 ターン 1 つ、**今季から即時適用**  
- 増産＝店在庫約1年分、増員、市場、厩舎、名馬、褒める  

### 5.5 市場

- 同季節の即転売は赤字（売＜買）  
- 季節またぎで利益  
- 価格は **季節×品目ごとの固定差分**（`seasonPriceDelta`）で決まる。店在庫の量には左右されない（旧・在庫比率連動の`priceMultMin/Max`方式は廃止・未使用）  
- 季節で店在庫増減＋`shopMin`  
- 完成品（剣・衝車・回復薬・弩・投石機・名剣・投刀・暗刀・雲梯・爆弾・火槍・火薬・しびれ薬・睡眠薬・仙丹）は売るだけ。レシピ・解禁条件・売値は[仕様書_実装版_v0_6_1.md §2.3](./仕様書_実装版_v0_6_1.md#23-生産)を参照

### 5.6 輸送

| ルート | 内容 |
|--------|------|
| 兵糧・兵器・武器 | 人足／荷車／馬車。船・馬荷役は廃止 |
| 騎馬次 | N0 手渡しのみ。出立後は **馬一覧から選択して自走**（手段なし） |
| 荷車 | 兵糧 N1 突破で解禁。同時使用上限 50 |
| 馬車 | 騎馬次 N2 かつ荷車解禁。速2・積載4・利用料50両。同時使用上限 100 |
| 人足 | 上限4。使用数＝ceil(荷量/1)。到着で復帰 |

完納すると **ノード+1・新依頼・駒移動**（`fix_node_advance_piece.js`）。

### 5.7 牧場

- 迷い馬・個体・季節サイクル（春仔・夏成長・秋つがい・冬餌）  
- 春の仔馬誕生はポップアップ  
- 秋以降、個体リストから売却可  
- 冬の結果文: 番がいれば「番になった馬に仔馬がうまれそうだ」

## 6. 変更の入れ方（実務）

### 数値・文章だけ

1. 正本 JSON / `game-data.js` を編集  
2. [データ同期とメンテナンス.md](./データ同期とメンテナンス.md) の該当コマンド  
3. Ctrl+F5・新規ゲーム確認  

### ロジック

1. `data/fix_自分の機能.js` を新規作成（既存を壊しにくい）  
2. HTML 末尾に IIFE で `window.xxx` を上書き  
3. マーカー文字列 `[xxx] ready` を入れる  
4. `node data/fix_自分の機能.js`  
5. 構文 OK・プレイ確認  

### 絶対にやってはいけないこと

- バックアップなしの `rm` や force 上書き  
- 本番 HTML を巨大 diff で直接手書き（失敗時に復旧不能）  
- 同じ関数を 3 重にラップして放置

## 8. 既知の注意・技術負債

1. **パッチの積み重ね**で同名 `window.xxx` が複数定義されている  
   → 最終定義が効く。壊れたら「最後の定義」を疑う  
2. 手紙・GAME_DATA が HTML に巨大埋め込み  
3. `data/sync_ui.js` と最新 fix の経路が一部二重  
4. Vite プロジェクト（`mou-isso/`）は未統合  
5. 馬の個体輸送は **騎馬次専用**（他皇子への馬運送は廃止）
